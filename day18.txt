MySQL VIEWS 
1. What Is a View? 
A view is a virtual table in MySQL, built from the result of a stored SELECT statement. It looks 
and behaves like a regular table when you query it — you can SELECT from it, filter it, and join 
it — but it does not store any data of its own. Every time a view is queried, MySQL runs its 
underlying SELECT statement against the real ("base") tables and returns the result. 
Think of a view as a saved, reusable query with a name. Instead of typing out a long JOIN every 
time, you create the view once, then simply run: SELECT * FROM view_name; 
CREATE VIEW view_name AS 
SELECT column1, column2, ... 
FROM table_name 
WHERE condition; 
2. Why Do We Need Views? 
●       Simplify complex queries: Hide multi-table JOINs, subqueries, and calculations 
behind a single, simple table-like name. 
●       Reuse query logic: Write a complex SELECT once and reuse it everywhere — 
reports, applications, dashboards — without copy-pasting SQL. 
●       Data security: Expose only specific columns or rows to certain users, without 
granting them access to the full base table. 
●       Different perspectives of the same data: Different departments can have their own 
tailored 'view' of the same underlying tables. 
●       Logical data independence: If the underlying table structure changes slightly, you 
can often adjust just the view definition and keep all dependent queries working 
unchanged. 
●       Simplified reporting: Pre-package aggregations (totals, averages, counts) into a 
view so business users can query them without knowing SQL joins or GROUP BY. 
3. Advantages of Views 
●       Security: restrict access to sensitive columns (e.g. salary) or sensitive rows (e.g. only 
a user's own department). 
●       Simplicity: end users and applications interact with a simple, named virtual table 
instead of complex SQL. 
●       Reusability: one view definition can be reused across many reports, applications, and 
ad-hoc queries. 
●       Consistency: business logic (calculations, filters) lives in one place, so results are 
always consistent. 
●       Maintainability: a business-rule change only needs to be made once, inside the view. 
●       Custom data presentation: rename or reformat columns, add computed columns, 
without touching the base table. 
●       No extra storage cost: a standard view stores no data — only its query definition. 
4. What Happens Without Views? 
Without views, every application, report, or analyst that needs the same 
joined/filtered/aggregated data must write out (and maintain) the full SQL logic independently. 
This leads to duplicated code, inconsistent results when the logic drifts between copies, harder 
security control, and painful maintenance whenever the business rules or schema change.

6. View Examples 
6.1  Basic View — Employee-Department Join 
Use case: avoid re-writing the JOIN between employees and departments every time someone 
needs an employee's department name. 
CREATE VIEW vw_employee_department AS 
SELECT e.emp_id, e.emp_name, e.designation, d.dept_name, d.location 
FROM employees e 
JOIN departments d ON e.dept_id = d.dept_id; 
  -- Usage 
SELECT * FROM vw_employee_department; 
6.2  Security View — Hiding Salary 
Use case: HR generalists should see employee details, but salary must stay restricted to payroll 
staff only. 
CREATE VIEW vw_employee_public AS 
SELECT emp_id, emp_name, designation, dept_id, join_date 
FROM employees; 
  -- Grant access to the view only, never to the base table 
GRANT SELECT ON company.vw_employee_public TO 'hr_user'@'%'; 
6.3  Aggregate / Reporting View — Department Salary 
Summary 
Use case: management wants total and average salary per department without re-writing GROUP 
BY logic every time. 
CREATE VIEW vw_department_salary_summary AS 
SELECT d.dept_name, 
    COUNT(e.emp_id)   AS total_employees, 
    SUM(e.salary)  AS total_salary, 
    AVG(e.salary)  AS average_salary 
FROM employees e 
JOIN departments d ON e.dept_id = d.dept_id 
GROUP BY d.dept_name; 
  -- Usage 
SELECT * FROM vw_department_salary_summary; 
Note: because this view uses GROUP BY and aggregate functions, it is read-only — it cannot be 
used for INSERT, UPDATE, or DELETE. 
6.4  Filtered, Updatable View — WITH CHECK OPTION 
Use case: an 'IT department only' view that stays consistent even when new rows are added 
through it. 
CREATE VIEW vw_it_employees AS 
SELECT emp_id, emp_name, designation, dept_id, salary 
FROM employees 
WHERE dept_id = 1 
WITH CHECK OPTION; 
  -- This succeeds (dept_id matches the view's WHERE clause) 
INSERT INTO vw_it_employees (emp_name, dept_id, designation, salary) 
VALUES ('Vijay', 1, 'DevOps Engineer', 58000); 
  -- This fails because dept_id = 2 violates the view's WHERE condition 
INSERT INTO vw_it_employees (emp_name, dept_id, designation, salary) 
VALUES ('Sneha', 2, 'HR Assistant', 40000); 
6.5  Computed Column View — Annual Salary 
Use case: present a calculated column (annual salary) without altering the base table. 
CREATE VIEW vw_employee_annual_salary AS 
SELECT emp_name, salary AS monthly_salary, salary * 12 AS annual_salary 
FROM employees; 
6.6  Multi-Level View — Built on Another View 
Use case: a 'high earners' view built directly on top of the department salary summary view. 
CREATE VIEW vw_high_paying_departments AS 
SELECT dept_name, average_salary 
FROM vw_department_salary_summary 
WHERE average_salary > 50000; 
 
  
7. Updating Data Through Views 
A view is updatable only when MySQL can unambiguously map each row of the view back to 
exactly one row of one base table. A view becomes non-updatable if it contains any of the 
following: 
●       Aggregate functions: SUM(), COUNT(), AVG(), MIN(), MAX() 
●       GROUP BY or HAVING 
●       DISTINCT 
●       UNION or UNION ALL 
●       Subqueries in the SELECT list that refer to the same base table 
●       Joins where the modified column doesn't map to a single base table -- Updatable example 
UPDATE vw_it_employees 
SET salary = 60000 
WHERE emp_name = 'Vijay'; 
  -- Reflected immediately in the base employees table 
8. Managing Views — ALTER, DROP, SHOW -- View the exact definition of a view 
SHOW CREATE VIEW vw_employee_department; 
  -- Modify an existing view's definition 
ALTER VIEW vw_employee_department AS 
SELECT e.emp_id, e.emp_name, d.dept_name 
FROM employees e 
JOIN departments d ON e.dept_id = d.dept_id; 
  -- Remove a view (base table data is untouched) 
DROP VIEW vw_employee_department; 
  -- List all views in the current database 
SELECT table_name 
FROM information_schema.views 
WHERE table_schema = 'jfs2'; 
 
  
9. Types of Views 
●       Simple View: Built from a single table, with no aggregate functions, GROUP BY, or 
joins. Usually fully updatable. 
●       Complex View: Built from multiple tables using joins, and/or includes aggregate 
functions, GROUP BY, or subqueries. Often read-only. 
●       Updatable View: A view (simple or a well-formed join) through which 
INSERT/UPDATE/DELETE can be performed. 
●       Read-Only View: Any view containing aggregates, GROUP BY, DISTINCT, or 
UNION — data can only be selected, never modified through it. 
●       Materialized View (not native to MySQL): A view whose results are physically 
stored and periodically refreshed. MySQL does not support this natively — it must be 
simulated using a real summary table refreshed via a scheduled EVENT or trigger. 
10. Interview Questions — Basic Level 
Q1. What is a view in MySQL? 
A view is a virtual table based on the result of a stored SELECT query. It does not store data itself (in standard 
MySQL views) — it stores the query definition, and the data is fetched from the underlying base table(s) each 
time the view is used. 
Q2. How do you create a view? 
Use the CREATE VIEW statement followed by a SELECT query. Example: CREATE VIEW view_name AS 
SELECT column1, column2 FROM table_name WHERE condition; 
Q3. How do you see the data inside a view? 
You query it exactly like a table: SELECT * FROM view_name; 
Q4. How do you check the underlying SQL definition of a view? 
Use SHOW CREATE VIEW view_name; — this returns the exact CREATE VIEW statement used to define it. 
Q5. How do you delete (remove) a view? 
Use DROP VIEW view_name; This removes only the view definition, never the underlying table data. 
Q6. Does a view store data physically? 
No. A standard MySQL view does not store data. It stores only the query. Every time the view is accessed, 
MySQL re-executes the underlying query against the base table(s). 
Q7. What is the main difference between a table and a view? 
A table physically stores data on disk, while a view is a stored query that presents data drawn live from one or 
more tables. Views have no independent storage for rows. 
Q8. Can a view be created using multiple tables? 
Yes. A view can be based on a JOIN across two or more tables, exactly as a normal SELECT query can. 
Q9. Can you use WHERE, GROUP BY, and ORDER BY inside a view definition? 
Yes, a view can include WHERE, GROUP BY, HAVING, JOIN, and (with some restrictions in subqueries) 
ORDER BY clauses, since it is built from any valid SELECT statement. 
Q10. Is it possible to rename a view? 
MySQL has no direct RENAME VIEW command. The common approach is to create a new view with the 
desired name using the same SELECT definition, then drop the old view. 
11. Interview Questions — Intermediate Level 
Q1. What is an updatable view? What restrictions apply? 
An updatable view allows INSERT, UPDATE, and DELETE operations that pass through to the base table. A 
view is updatable only if it has no aggregate functions (SUM, COUNT, AVG, etc.), no GROUP 
BY/HAVING/DISTINCT, no UNION, and comes from a single table (or a join where the modified columns 
map to one underlying table). 
Q2. What is the WITH CHECK OPTION and why is it used? 
WITH CHECK OPTION ensures that any INSERT or UPDATE performed through the view still satisfies the 
view's WHERE clause. Without it, a user could insert or update a row through a filtered view and the row 
would then silently disappear from that view's result set. 
Q3. Can you insert data into a view? Under what conditions? 
Yes, but only if the view is updatable (see above) and the INSERT provides values for all NOT NULL columns 
of the base table that are not included in the view (or those columns have defaults). 
Q4. Does MySQL support materialized views? 
Not natively. MySQL views are always virtual — they run the underlying query on every access. To get 
materialized-view-like behavior, you must manually create a real table and refresh it periodically (e.g. via a 
scheduled event or trigger), which acts as a workaround. 
Q5. Can a view be created on top of another view? 
Yes, MySQL allows nested views — a view can be defined using a SELECT that reads from another view. 
However, deeply nested views can hurt performance and make debugging harder. 
Q6. What happens if you drop or alter a base table that a view depends on? 
MySQL does not check view dependencies at DROP time. If you drop or rename a column/table used by a 
view, the view becomes invalid, and querying it will raise an error at execution time (MySQL uses a 'late 
binding' approach for views). 
Q7. Do views improve query performance? 
Not directly. A view is essentially a saved query — MySQL still executes the full underlying SELECT (join, 
filters, aggregation) every time the view is queried. Views improve readability, reusability, and security, not 
raw execution speed. In fact, deeply nested or poorly written views can slow things down. 
Q8. How do you modify an existing view's definition? 
Use ALTER VIEW view_name AS SELECT ...; to redefine it, or DROP VIEW followed by CREATE VIEW 
with the new definition. 
Q9. Can indexes be created directly on a MySQL view? 
No. Since a standard MySQL view has no physical storage, you cannot create an index directly on it. 
Performance for a view depends entirely on the indexes that exist on its underlying base tables. 
Q10. Can a view include an ORDER BY clause, and does it always take effect? 
A view can include ORDER BY, but if the outer query that selects from the view also has its own ORDER BY, 
the outer ORDER BY takes precedence, and the view's internal ordering may be ignored by the optimizer. 
12. Interview Questions — Advanced Level 
Q1. How do views help implement security and access control? 
Views let you expose only specific columns or rows of a table to certain users (column-level and row-level 
security), without granting them direct access to the base table. For example, a view can omit a salary column 
entirely so HR staff can see employee details but not compensation data, and you grant SELECT only on the 
view, not on the table. 
Q2. What are the key limitations of MySQL views? 
Views cannot have their own indexes; they cannot use certain constructs (like variables, some subquery forms) 
that make them non-updatable; performance depends on the underlying query being re-run every time; and 
MySQL does not validate view dependencies automatically when base tables change. 
Q3. Explain MySQL's ALGORITHM=MERGE vs ALGORITHM=TEMPTABLE for 
views. 
MERGE algorithm rewrites the outer query and the view's query into a single query and pushes conditions 
down to the base tables — this is generally faster and allows the view to be updatable. TEMPTABLE 
algorithm executes the view's query first into an internal temporary table, then runs the outer query against that 
temp table — this is often slower and makes the view non-updatable. MySQL uses MERGE whenever 
possible and falls back to TEMPTABLE when the view contains aggregates, DISTINCT, GROUP BY, 
UNION, or subqueries in the SELECT list. 
Q4. How would you find out which views depend on a specific table? 
Query the information_schema.VIEWS table and inspect the VIEW_DEFINITION column (or use 
information_schema.TABLES together with view metadata) to search for references to the table name. There 
isn't a single built-in 'dependency' table, so text-searching VIEW_DEFINITION for the table name is the 
common approach: SELECT table_name, view_definition FROM information_schema.views WHERE 
view_definition LIKE '%employees%'; 
Q5. Can a view span multiple databases in MySQL? 
Yes. A view's SELECT statement can reference tables from a different database using db_name.table_name 
syntax, as long as the user creating/using the view has the right privileges on both databases. 
Q6. Describe a scenario where a view could hurt performance. 
A reporting view that joins several large tables with aggregate functions, built on top of two or three other 
nested views, forces MySQL to use the TEMPTABLE algorithm and materialize large intermediate result sets 
in memory or on disk for every single query — even a simple SELECT * FROM view WHERE emp_id = 5 
ends up scanning and joining much more data than necessary. 
Q7. What is the DEFINER clause in a view, and why does it matter? 
When a view is created, MySQL records a DEFINER (the user account that created it). By default, the view 
executes with the privileges of that definer, not the querying user (SQL SECURITY DEFINER, the default). 
This lets you grant end users access to a view without granting them any direct privileges on the underlying 
base tables — the view runs with the definer's higher privileges. SQL SECURITY INVOKER changes this so 
the view runs with the querying user's own privileges instead. 
Q8. Can views support aggregate functions and GROUP BY? Are such views updatable? 
Yes, views can include SUM(), COUNT(), AVG(), GROUP BY, and HAVING. However, such views are never 
updatable — MySQL disallows INSERT/UPDATE/DELETE through any view containing aggregate 
functions, GROUP BY, HAVING, DISTINCT, or UNION, because there's no unambiguous way to map a 
change back to a single base-table row. 
Q9. How can you achieve materialized-view-like behavior in MySQL, since it isn't 
supported natively? 
Create a real physical 'summary' table that mirrors the view's result, and populate/refresh it periodically using a 
MySQL EVENT (scheduled job) or a trigger-driven update. Applications query this summary table directly for 
fast reads, while the refresh job keeps it in sync with the source tables — trading real-time accuracy for read 
performance. 
Q10. If two views are nested (a view built on another view), how does MySQL execute the 
outer query? 
MySQL expands nested views at parse time — the outer view's query is effectively rewritten to substitute the 
inner view's SELECT definition, then the whole thing is optimized together where possible using the MERGE 
algorithm. If any level requires TEMPTABLE (e.g. due to aggregation), MySQL materializes that intermediate 
result first, which can compound performance costs across multiple nesting levels. 
13. Real-World Scenario-Based Questions 
These questions are commonly asked in interviews to test whether a candidate understands 
practical, real-world use of views rather than just syntax. 
Scenario 1. HR staff need to see employee names, designations, and departments, but must 
NOT see salary information. How would you solve this using views? 
Solution: Create a view that selects only the non-sensitive columns (emp_id, emp_name, designation, dept_id) 
and excludes salary entirely. Grant HR staff SELECT privilege only on this view — not on the base employees 
table. They can never query a column that was never included in the view's definition. 
Scenario 2. Management wants one query that always shows total and average salary per 
department, without every analyst having to write the same JOIN + GROUP BY each time. 
How do views help? 
Solution: Create a reporting view (e.g. vw_department_salary_summary) that performs the JOIN between 
employees and departments with SUM(salary) and AVG(salary) grouped by department, once. Analysts then 
simply run SELECT * FROM vw_department_salary_summary; getting consistent, up-to-date figures without 
duplicating the underlying logic in every report or tool. 
Scenario 3. A 5-table join with complex business logic is copy-pasted into 20 different places 
in the application. What problem does this cause, and how do views fix it? 
Solution: Duplicating the same complex query in 20 places means every future business-rule change (a new 
filter, a renamed column, an extra join condition) has to be updated in 20 places, which is error-prone and 
inconsistent. Wrapping the logic in a single view centralizes it — the application code everywhere simply does 
SELECT ... FROM the_view, and any future change to the business logic is made once, inside the view 
definition. 
Scenario 4. The DBA wants to restrict staff to only certain columns of a sensitive table, 
without managing column-level GRANTs. How do views solve this? 
Solution: Create a view exposing only the approved columns, and grant privileges on the view instead of the 
base table. This achieves column-level security without MySQL's more cumbersome column-level GRANT 
syntax, and it's easy to audit since all restricted access flows through one well-named view. 
Scenario 5. A view is created with WITH CHECK OPTION and a WHERE clause restricting 
it to department = 'IT'. A user tries to INSERT a row through this view with department = 
'HR'. What happens? 
Solution: The INSERT is rejected by MySQL with an error, because WITH CHECK OPTION enforces that 
any row inserted or updated through the view must still satisfy the view's WHERE condition (department = 
'IT'). Without WITH CHECK OPTION, the insert would silently succeed in the base table but the new HR row 
would simply not appear when querying the view again. 
Scenario 6. A view was created using ALGORITHM=TEMPTABLE because it contains a 
GROUP BY. What's the practical impact for the team using it? 
Solution: The view will always be read-only (no INSERT/UPDATE/DELETE through it), and every SELECT 
against it forces MySQL to build an internal temporary table first — which is slower than the MERGE 
approach, especially on large base tables. For a frequently queried report, the team should consider 
periodically caching the results in a real summary table instead of hitting this view constantly. 
Scenario 7. A column referenced by a view gets renamed by another team during a migration. 
What happens to the view, and how would you catch this early? 
Solution: MySQL does not validate views against schema changes at the time the column is renamed — the 
view keeps its old definition text. The problem only surfaces when someone actually queries the view and gets 
an 'Unknown column' error. To catch this early, teams should search information_schema.views for references 
to any column/table before making a breaking schema change, or add automated tests that run SELECT * 
FROM view_name; against a staging database as part of the migration checklist. 
Scenario 8. Your reporting dashboard needs fast reads on aggregated sales data, but MySQL 
views don't cache or materialize results. How would you work around this? 
Solution: Build a real, physical 'materialized view' equivalent: a summary table (e.g. daily_sales_summary) 
populated by a scheduled MySQL EVENT or an ETL job that runs the aggregate query and stores the results. 
The dashboard reads from this fast, indexed summary table instead of recomputing the aggregation on every 
page load, accepting a small refresh-interval delay in exchange for speed.

1. Single Row Subqueries 
A single-row subquery returns exactly one column and one row, so the 
outer query can compare against it directly using =, >, <, >=, <=, or <>. 
These are usually built with aggregate functions (MAX, MIN, AVG, 
SUM, COUNT) or a WHERE clause specific enough to isolate one row. 
General syntax: 
SELECT columns FROM outer_table 
WHERE column [=|>|<|>=|<=|<>] (SELECT single_value FROM inner_table [WHERE ...]);

2. Multi-Row Subqueries — ANY / SOME 
ANY (and its synonym SOME) compares a value against a LIST of 
values returned by the subquery. The condition is TRUE if it holds for 
AT LEAST ONE row in that list. > ANY means 'greater than the 
smallest value in the list'; < ANY means 'less than the largest value in 
the list'. 
General syntax: 
SELECT columns FROM outer_table 
WHERE column [=|>|<|>=|<=] ANY (SELECT column FROM inner_table [WHERE ...]);

3. Multi-Row Subqueries — ALL 
ALL compares a value against every row returned by the subquery. The 
condition is TRUE only if it holds for EVERY row. > ALL means 
'greater than the largest value in the list'; < ALL means 'less than the 
smallest value in the list'.

SELECT columns FROM outer_table 
WHERE column [=|>|<|>=|<=|<>] ALL (SELECT column FROM inner_table [WHERE ...]);

4. Nested Subqueries 
A nested subquery is a subquery inside another subquery — the 
inner-most query resolves first, its result feeds the next level up, and so 
on, until the outermost query runs last. These subqueries usually do 
NOT reference the outer query's columns (uncorrelated). 
General syntax: 
SELECT columns FROM t1 
WHERE col IN (SELECT col FROM t2 
           WHERE col2 IN (SELECT col2 FROM t3 WHERE ...));

5. Co-Related (Correlated) Subqueries 
A correlated subquery references a column from the OUTER query 
inside its WHERE clause, so it is conceptually re-evaluated once per 
outer row (e.g. 'employees earning more than the average of their OWN 
department'). This is the pattern behind classic 'top-N-per-group' queries. 
General syntax: 
SELECT columns FROM t1 outer 
WHERE outer.col [operator] (SELECT AGG(col) FROM t2 inner 
                         WHERE inner.group_col = outer.group_col);

6. EXISTS and NOT EXISTS 
EXISTS returns TRUE if a (usually correlated) subquery returns at least 
one row; NOT EXISTS returns TRUE when it returns none. The actual 
columns selected inside don't matter — only row presence — which is 
why SELECT 1 is the convention. 
General syntax: 
SELECT columns FROM t1 
WHERE [NOT] EXISTS (SELECT 1 FROM t2 WHERE t2.fk = t1.pk [AND ...]);

Advanced Level  (10 questions) 
Q1. Find departments where EVERY employee has been assigned to at least one project (nested NOT 
EXISTS = 'for all'). 
SELECT DeptName FROM department d 
WHERE NOT EXISTS (SELECT 1 FROM employee e 
               WHERE e.DeptID = d.DeptID 
                 AND NOT EXISTS (SELECT 1 FROM employee_project ep WHERE ep.EmpID = e.EmpID)); 
Verified Result: 
(no rows returned) 
  
Q2. Find departments where at least one employee has NOT been assigned to any project (EXISTS 
wrapping NOT EXISTS). 
SELECT DISTINCT d.DeptName FROM department d 
WHERE EXISTS (SELECT 1 FROM employee e 
           WHERE e.DeptID = d.DeptID 
             AND NOT EXISTS (SELECT 1 FROM employee_project ep WHERE ep.EmpID = e.EmpID)); 
Verified Result:

Finance 
  
Q3. Find managers whose ENTIRE team earns more than the company average salary (NOT EXISTS = 
no underperformer). 
SELECT DISTINCT m.Name FROM employee m 
WHERE EXISTS (SELECT 1 FROM employee e WHERE e.ManagerID = m.EmpID) 
  AND NOT EXISTS (SELECT 1 FROM employee e 
               WHERE e.ManagerID = m.EmpID 
                 AND e.Salary <= (SELECT AVG(Salary) FROM employee)); 
Verified Result: 
Name 
John Smith

Q5. Find departments that run at least one project but have no project with a budget below 100000 
(EXISTS + NOT EXISTS combined). 
SELECT DeptName FROM department d 
WHERE EXISTS (SELECT 1 FROM project p WHERE p.DeptID = d.DeptID) 
  AND NOT EXISTS (SELECT 1 FROM project p WHERE p.DeptID = d.DeptID AND p.Budget < 100000); 
Verified Result: 
DeptName 
Sales 
Engineering 
  
Q6. Find sales regions where every sales rep active in that region has made at least 3 sales (nested NOT 
EXISTS 'for all reps'). 
SELECT DISTINCT s.Region FROM sales s 
WHERE NOT EXISTS (SELECT 1 FROM sales s2 
               WHERE s2.Region = s.Region 
                 AND (SELECT COUNT(*) FROM sales s3 
                      WHERE s3.EmpID = s2.EmpID AND s3.Region = s2.Region) < 3); 
Verified Result: 
Region 
North 
South 
East 
West 
  
Q7. Find employees who earn more than every one of their department colleagues who is assigned to a 
project (EXISTS + correlated comparison). 
SELECT e1.Name FROM employee e1 
WHERE EXISTS (SELECT 1 FROM employee_project ep0 WHERE ep0.EmpID = e1.EmpID) 
  AND NOT EXISTS (SELECT 1 FROM employee e2 
               WHERE e2.DeptID = e1.DeptID 
                 AND EXISTS (SELECT 1 FROM employee_project ep WHERE ep.EmpID = e2.EmpID) 
                 AND e2.Salary > e1.Salary); 
Verified Result:

Q8. Find departments where at least one project has a budget larger than every employee's salary in 
that department (EXISTS + NOT EXISTS 'greater than all'). 
SELECT DISTINCT d.DeptName FROM department d 
WHERE EXISTS (SELECT 1 FROM project p 
           WHERE p.DeptID = d.DeptID 
             AND NOT EXISTS (SELECT 1 FROM employee e 
                             WHERE e.DeptID = d.DeptID AND e.Salary >= p.Budget)); 
Verified Result: 
DeptName 
Engineering 
  
Q9. Find employees who are the ONLY person in their department assigned to any project (no colleague 
in the same department is also assigned). 
SELECT e1.Name FROM employee e1 
WHERE EXISTS (SELECT 1 FROM employee_project ep WHERE ep.EmpID = e1.EmpID) 
  AND NOT EXISTS (SELECT 1 FROM employee e2 
               JOIN employee_project ep2 ON e2.EmpID = ep2.EmpID 
               WHERE e2.DeptID = e1.DeptID AND e2.EmpID <> e1.EmpID); 
Verified Result: 
(no rows returned) 
Q10. Find managers all of whose direct reports have made at least one sale (nested NOT EXISTS 'for all
